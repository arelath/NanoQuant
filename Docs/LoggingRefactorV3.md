# Logging and Run Discovery V3

**Status:** Proposed

**Audience:** Maintainers, tooling engineers, and algorithm researchers

**Alternative to:** [LoggingRefactorV2.md](LoggingRefactorV2.md)

**Related:** [LoggingRefactor-Review.md](LoggingRefactor-Review.md), [07-observability-and-reporting.md](07-observability-and-reporting.md), [15-performance-profiling.md](15-performance-profiling.md), [16-behavior-preserving-optimizations.md](16-behavior-preserving-optimizations.md)

## 1. Summary

This design gives NanoQuant a single event-routing pipeline with enforced severity thresholds, real run identity for the resident research workflows, a discovery registry of immutable facts, and a CLI that can list, resolve, and follow any run. It satisfies every acceptance criterion in [LoggingRefactor-Review.md](LoggingRefactor-Review.md) §5.

It differs from V2 in one governing principle: **derived views are computed, not maintained.** V2 stores a writer-maintained `run.log`, a lifecycle-snapshot index, and a `latest.json` pointer — three derived files, each with its own staleness, reconciliation, and crash-recovery contract. V3 stores no mutable derived state at all. Human-readable run and memory views are rendered from `events.jsonl` whenever they are needed; the discovery registry records only facts that never change; `latest` is a query, not a file. Anything that is computed on demand cannot be stale, cannot be ahead of its source, and cannot need repair.

The costs of that principle are quantified in §13 and accepted deliberately.

## 2. Design principles

1. **One creator, many destinations.** Exactly one component validates an event, assigns its sequence and timestamp, and constructs the immutable envelope. Destinations only receive envelopes.
2. **Canonical evidence is a superset by construction.** Every derived view (console, CLI rendering, saved `run.log` and `memory.log`) is a filtered rendering of `events.jsonl`. There is no configuration that can make a derived view more verbose than the canonical stream, so no invariant between three thresholds needs policing.
3. **Store immutable facts; read live state from its owner.** The registry records `(run_id, path, created_at)` — facts that are true forever. Status, `updated_at`, and artifacts are always read from `manifest.json`, which is already atomically replaced and authoritative. Nothing in the discovery layer can go stale.
4. **Optional observability must not touch the algorithm.** A console or rendering failure never changes stage results, committed artifacts, or `state/journal.jsonl`. Canonical event-write failure is fatal because the audit contract is broken.
5. **Smallest possible configuration surface.** This design adds **zero** fields to `ObservabilityConfig`. It makes the existing `console_level` and `event_level` effective and validated. Strict decoding, canonical serialization, `config_hash`, and old recipes are untouched.

## 3. Sources of truth

| Data | Purpose | Authority | Staleness/recovery |
|---|---|---|---|
| `manifest.json` | Run identity, lifecycle, provenance, config | Authoritative | Atomic replace (existing behavior) |
| `events.jsonl` | Ordered diagnostic/audit events | Canonical event stream | Valid prefix readable; torn tail quarantined at writer open (§6.6) |
| `state/journal.jsonl` | Validated commit boundaries for resume | Authoritative resume stream | Existing `ProgressJournal` rules; logging code never touches it |
| `.nanoquant/registry.jsonl` | Pointers to runs outside the run root | Rebuildable pointer list | Torn lines skipped; `runs rebuild-registry` rewrites |
| `run.log` | One-shot progress/decision view without memory meter noise | Disposable output | Regenerated by `logs --save` or at session close; safe to delete |
| `memory.log` | One-shot memory samples and memory fields from boundary events | Disposable output | Regenerated with `run.log`; safe to delete |
| `profile.json` / `profile.md` | Performance evidence | Profiling design | Out of scope here |

The word *journal* refers exclusively to `state/journal.jsonl`. CLI names, help text, and code identifiers must never call `events.jsonl` or its renderings a journal. `runs path --kind journal` resolves the progress journal explicitly; `logs` resolves the event view.

## 4. Current-state facts this design is built on

- `JsonlEventSink` ([events.py](../src/nanoquant/infrastructure/events.py)) creates envelopes, assigns sequences by scanning the whole file at open, serializes with `default=str`, flushes without `fsync` while its docstring claims durability, and calls one observer whose exceptions propagate into the caller.
- `bootstrap.run_experiment` is the only path that creates a `RunDirectory`, manifest, and lease. `resident_quantization.py` (twice) and `tiny_pipeline.py` construct `JsonlEventSink` directly in caller-selected directories with the constant run IDs `resident-quantization`, `resident-factorization-slice`, and `tiny-run`, with no manifest and no lease.
- `RunManifest.resolved_config` is already `dict[str, object]`, not `RunConfig` — the same manifest dataclass can describe resident runs without a second schema.
- `ObservabilityConfig.console_level` and `event_level` are unvalidated strings that nothing reads.
- `capture_environment` already provides allowlisted, secret-redacted environment capture with `SECRET_PATTERN`.
- `RunLease` is create-exclusive with pid metadata and no staleness handling: a crashed process blocks resume until the lease file is deleted by hand.
- The CLI has only `inspect` and `config-reference`.
- All current emitters use severities `info`, `warning`, `error`.

## 5. Severity

A four-member ordered enum:

```text
debug < info < warning < error
```

Rationale: nothing in the codebase emits below `info` or above `error` today; `record_admm_steps` is the only planned sub-info producer and `debug` covers it. `trace` and `critical` are not added until a concrete producer exists — extending an enum later is an additive, compatible change, while carrying unused levels means testing and documenting behavior nobody exercises.

Rules:

- `Severity` is a `str`-valued enum in `ports/event_sink.py`; the `Event` envelope keeps its `severity: str` field and existing schema version 1, so old streams remain readable.
- `config/validation.py` gains checks (new `OBS0xx` codes):
  - `OBS001`: `console_level` / `event_level` must name a valid severity.
  - `OBS002`: `event_level <= console_level` (the canonical stream must be at least as verbose as the console). With computed views this is the only threshold pair that exists.
  - `OBS003`: `record_admm_steps: true` requires `event_level: debug` — otherwise the flag is a silent no-op, which is worse than an error.
- Validation is behavior only, not serialization: `config_hash`, canonical config output, and strict decoding are unchanged.

## 6. Event pipeline

### 6.1 EventRouter

`EventRouter` implements the existing `EventSink` protocol and replaces the envelope-creation role of `JsonlEventSink`:

```python
class EventRouter:
    def __init__(
        self,
        run_id: str,
        *,
        event_level: Severity,
        destinations: tuple[EventDestination, ...],
        initial_sequence: int = 0,
    ) -> None: ...

    def emit(self, stage, severity, name, *, span_id=None, parent_span_id=None, **fields) -> Event | None: ...
    def span(self, stage, name, parent_span_id=None, **fields) -> Iterator[str]: ...
    def close(self) -> None: ...
```

`emit` performs, in order:

1. validate stage/name identifiers and parse severity (programmer errors raise immediately — these are caught by tests, not production);
2. if `severity < event_level`, return `None` — the event is never created, sequenced, or delivered anywhere;
3. sanitize and redact fields (§6.4) — never raises;
4. under the per-process lock, assign the next sequence and UTC timestamp and build one immutable `Event`;
5. deliver to destinations in declaration order, required destinations first;
6. return the envelope.

The `EventSink` protocol return type changes to `Event | None`. Callers almost universally ignore the return; tests that assert on returned envelopes run with `event_level: debug`. Suppressing below `event_level` at creation (rather than at each destination) keeps sequences contiguous in `events.jsonl` and makes "what is in the canonical stream" a single-knob question.

### 6.2 Destinations

```python
class EventDestination(Protocol):
    role: Literal["required", "optional"]
    threshold: Severity
    def accept(self, event: Event) -> None: ...
    def flush(self) -> None: ...
    def close(self) -> None: ...
```

Initial destinations:

- `JsonlEventDestination` — required, threshold = `event_level` (accepts everything routed). Writes compact JSON with `sort_keys=True`, `allow_nan=False`, and **no** `default=` hook; the sanitizer guarantees serializability before this point. Flushes per event (no `fsync`).
- `ConsoleEventDestination` — optional, threshold = `console_level`. Renders ordinary events with the single shared
  line renderer (§8.1), folds `calibration.progress_*` events into a console-only `Calibrating` batch bar, and then
  folds resident `compression.progress_initialized`, block, and layer events into the `Compressing Layers`
  progress/elapsed/ETA line. A TTY redraws one carriage-return line; captured output receives bounded start/finish
  checkpoints. Resume seeds compression elapsed time and ETA from durable completed-block results, without reading
  the historical event stream. The progress-bar text is not part of `run.log` or `memory.log`.

There is no text-file destination; see §8. The destination protocol deliberately supports adding one (or a remote forwarder) later without touching the router.

### 6.3 Failure policy

- A **required** destination exception is wrapped in `EventWriteError` and propagates. The run must stop: canonical evidence can no longer be written.
- An **optional** destination exception quarantines that destination for the remainder of the process. The router then emits one `observability.destination_disabled` event (severity `warning`, fields `destination`, `error_type`) through the surviving destinations, guarded by a re-entrancy flag so a second failure cannot recurse.
- Required destinations are delivered **before** optional ones, so the console never displays an event that failed to reach `events.jsonl`.
- Tests must prove (§14) that a fault-injected console failure produces byte-identical committed artifacts and journal contents for a deterministic fixture.

### 6.4 Field contract

A `sanitize_fields` function applied once per emit:

- **Allowed values:** `None`, `bool`, `int`, finite `float`, `str`, and lists/dicts of the same, to depth 4. Dict keys are coerced to `str`.
- **Non-finite floats** become the strings `"nan"`, `"inf"`, `"-inf"`. Sanitization never raises: a malformed diagnostic value must not kill a 30-hour run. Anything outside the allowed types is replaced by `{"unsupported_type": "<qualified type name>"}`.
- **Caps:** a single string field is truncated at 4 KiB with a `…[truncated N bytes]` suffix; if a serialized event would exceed 32 KiB, oversized fields are replaced by truncation markers until it fits. Tensors, arrays, optimizer state, and full configs belong in artifacts, referenced by ID.
- **Redaction:** the existing `SECRET_PATTERN` from [environment.py](../src/nanoquant/infrastructure/environment.py) is applied to field keys and string values; matches become `<redacted>`. This reuses the tested pattern rather than inventing a second policy.
- **Determinism:** keys are sorted at serialization (already the case) and identifiers use lowercase dotted/snake-case names with units in the field name, per the observability design.

### 6.5 Durability

Console output is immediate. `events.jsonl` is flushed per event so `logs --follow` sees progress promptly, but it does **not** claim power-loss durability and never calls `fsync` per event — the incorrect docstring on `JsonlEventSink` is corrected. Resume correctness comes from `state/journal.jsonl` and committed artifacts, which keep their existing validate-then-`fsync` semantics. This split is stated in CLI help.

### 6.6 Sequence continuation and torn-tail recovery

One process owns the event stream for a run session, enforced by the run lease (§7.3). At writer open, under the lease:

1. read backward from EOF (bounded chunks, not a full-file scan — the current `_read_last_sequence` is O(history) and gets worse every resume) to find the last complete line;
2. if trailing bytes do not parse as a complete JSON line (torn write from a crash), append the raw bytes to `events.jsonl.orphan`, truncate the tail in place, and emit `observability.tail_recovered` with the quarantined byte count;
3. continue from `last valid sequence + 1`.

The recovery is automatic and deterministic because it happens only under the lease and only ever removes a provably incomplete final line. Readers (CLI, report rendering) never modify the file; they read the valid prefix and report a torn tail if present.

### 6.7 Compatibility wrapper

`JsonlEventSink` remains as a thin deprecated wrapper that builds an `EventRouter` with a single JSONL destination and `event_level: debug`, preserving its public `emit`/`span`/`close`/context-manager surface so existing tests and narrow callers migrate gradually. New composition code uses `EventRouter` via the run session (§7).

## 7. Run sessions and identity

### 7.1 `open_run_session`

One composition-root helper replaces scattered sink construction:

```python
@contextmanager
def open_run_session(
    output: Path,
    *,
    manifest: RunManifest,
    observability: ObservabilityConfig,
    registry_root: Path | None,
    console: bool = True,
) -> Iterator[RunSession]:
    ...

@dataclass(frozen=True, slots=True)
class RunSession:
    run_id: str
    output: Path
    events: EventSink
```

Responsibilities at open: create the directory, write the manifest atomically (or adopt the existing one on resume), acquire the lease, recover the event tail, build the router, and register with the registry when `registry_root` is set and `output` is not already a child of it. At close (in `finally`): flush and close all destinations idempotently, then render each of `run.log` and `memory.log` atomically from the same canonical prefix (§8.3).

The session owns observability and identity only. Manifest lifecycle transitions, artifact commits, and report rendering remain the caller's, exactly as `bootstrap.run_experiment` does today — the session must not become a second lifecycle authority.

Registration failure and final-render failure are warnings, never fatal: discovery and convenience degrade, the run does not.

### 7.2 Resident runs get real manifests — with the existing schema

`RunManifest.resolved_config` is already a plain dict, so resident runs need **no** parallel `ResearchRunManifest` schema. A resident session builds a `RunManifest` directly:

- `run_id`: `create_run_id()` for a fresh directory; read from the existing `manifest.json` on resume. The strings `resident-quantization` and `resident-factorization-slice` become the `component` key inside `resolved_config` and remain stage names in events — they are no longer run IDs.
- `resolved_config`: the canonical dict of the resident request plus `{"component": ..., "experiment_number": ...}` when known.
- `config_hash`: sha256 over that canonical JSON.
- `launcher`: `launcher_provenance(...)` with `arguments=()` — raw argv is never captured by default (review §3.8); an explicit allowlist can be added later if needed.
- `environment`: `capture_environment()`.
- Status transitions reuse the existing `transition()` state machine.

Resume in a caller-selected directory: if `manifest.json` exists, keep its `run_id`, transition `INTERRUPTED → RUNNING`, and emit `run.resumed` carrying both the stored and current `config_hash`. A hash mismatch is a **warning event**, not a failure — resident canary requests legitimately vary knobs between resumptions, and the numerical resume safety already lives in the progress journal's identity checks, not here.

Legacy resident directories whose events carry the old constant run IDs: the new session keeps the event sequence, uses the manifest's (new) run ID for new events, and records `previous_run_id` in `run.resumed`. Renderers do not correlate on `run_id`.

### 7.3 Leases

Direct resident entry points and the tiny pipeline acquire the same per-run lease that managed runs use, before opening the router. The lease file gains `hostname` next to the existing `pid`/`acquired_at`. On acquire conflict:

- same host and pid provably not alive → take over, emitting `run.lease_taken_over` with the previous owner;
- otherwise raise `RUN001` including owner metadata so the error is actionable instead of "file exists".

This fixes the current behavior where any crash leaves a lease that blocks resume until someone deletes a hidden file.

## 8. Human-readable views: computed, not maintained

### 8.1 One renderer, one line format

A pure projection selects the run, memory, or complete view, then the single `render_event_line(event) -> str` function is used by the console destination, `nanoquant logs`, and both text snapshots:

```text
2026-07-14T16:12:33.482Z 0001842 INFO    quantize-blocks block.completed block=5 loss=3.5971968174 wall_seconds=421.8
```

- UTC RFC 3339 timestamp normalized to `Z` form (the stored envelope keeps Python's `isoformat` output; normalization is display-only);
- zero-padded 7-digit sequence, fixed-width uppercase severity, stage, event name;
- fields sorted by key, compact JSON escaping, control characters escaped — exactly one physical line per event, always;
- no locale-dependent formatting.

The ordinary run projection removes `resource.sample`/host-cache maintenance records and strips `cuda.*`, `host.*`, `wddm.*`, peak, workspace, plan, and budget-utilization fields from mixed lifecycle events. The memory projection retains those records and fields plus correlation context such as block, layer, epoch, and attempt. OOM and sampler warnings remain visible in `run.log` with their non-meter diagnostic fields and are also present in `memory.log`. The canonical event is never mutated.

Because console, live follow, and archived text all go through the same projection and renderer, a researcher sees one vocabulary everywhere, and there is exactly one renderer to test for tail-safety and Unicode.

### 8.2 `nanoquant logs`

- `logs SELECTOR` renders the clean run view of the resolved event stream to stdout.
- `--memory` selects the memory view, including with `--follow`; this is the supported live memory-monitoring path.
- `--follow` renders existing events, then polls (default 1 s, configurable flag) for appended **complete newline-terminated** lines, tolerating temporary EOF and detecting truncation/replacement by re-syncing from the last rendered sequence. Ctrl+C exits cleanly without touching run state. When the manifest reaches a terminal status and the tail is drained, follow prints the terminal status and exits 0.
- `--level L` is a display filter; it can only hide events, never add ones absent from the canonical stream.
- `--json` emits canonical event objects unmodified unless `--memory` explicitly selects the projected memory view.
- `--save` writes each of `<run>/run.log` and `<run>/memory.log` atomically (temp file + `os.replace`).

### 8.3 Text logs as one-shot snapshots

There is no incrementally-maintained text file. `run.log` and `memory.log` exist in exactly two ways:

1. **At session close**, the session renders each projection atomically — so every finished run's directory contains browsable progress and memory logs without any tool.
2. **On demand**, via `logs --save`.

Both paths use the same complete canonical prefix. Consequences:

- there is no reconciliation state machine (absent/behind/ahead/corrupt/version), no `.run-log-state.json` sidecar, no crash window between a JSONL write and a text write, no resume catch-up logic, and no possibility of a derived line existing without its canonical event;
- a run that dies without reaching session close may have neither text log until someone runs `logs --save`; live inspection still reads `events.jsonl`;
- a resumed run's stale text logs are overwritten wholesale at the next close. Deleting either is always safe.

Cost: `Get-Content -Wait run.log` / `tail -f memory.log` does not work **during** a run; live monitoring goes through `nanoquant logs latest --follow` or `nanoquant logs latest --memory --follow` (or reading `events.jsonl` directly, which is line-oriented JSON). This trade is accepted: one supported follow path that is tested on Windows beats additional write paths that must be reconciled forever. Render cost is linear in event count and runs at file-I/O speed; even a million-event stream renders in seconds, and default-verbosity runs are orders of magnitude smaller.

## 9. Discovery: registry of immutable facts

### 9.1 Two complementary mechanisms

1. **Scanning.** Managed runs live at `<run_root>/<run_id>/manifest.json`. Listing scans the run root's direct children and reads their manifests. No registration step exists for them at all — the filesystem layout is already the index, and reading a few hundred small JSON files is comfortably interactive on local disks. If manifest size ever makes this slow, an mtime-keyed cache can be added later without changing any contract; it is explicitly not in this iteration.
2. **Registry.** Runs in caller-selected directories (`evidence/m4/...`) cannot be found by scanning. Their sessions append one line to `<run_root>/.nanoquant/registry.jsonl`:

```json
{"schema_version": 1, "run_id": "run_20260714T161233482000_ab12cd34", "path": "../evidence/m4/gemma-v24-canary", "created_at": "2026-07-14T16:10:00Z", "component": "resident-quantization", "experiment_number": 24}
```

Every field is immutable at registration time. Status, `updated_at`, artifacts, and failure details are **always** read from the target manifest at query time — the manifest wins by construction because the registry never stores anything the manifest owns. There are no lifecycle snapshots, no folding, no compaction, and nothing that can go stale. The file grows one line per external run, which is negligible.

Paths are stored root-relative when possible, absolute-normalized otherwise. Resolution rejects records whose resolved manifest is missing or whose `run_id` does not match (shown as `missing`/`mismatched` in `runs list`, skipped by selectors), which also neutralizes hand-edited traversal paths.

### 9.2 Concurrency

Registration takes `<run_root>/.nanoquant/registry.lock` (create-exclusive with owner metadata, bounded retry ~5 s, stale takeover after 60 s with a warning), appends one flushed line, and releases. Readers never take the lock: they parse the file skipping malformed lines, deduplicate by `(run_id, path)` keeping the first occurrence, and verify manifests. `runs rebuild-registry [--include-root PATH]...` scans configured roots plus revalidated absolute entries from the existing file and rewrites it atomically.

### 9.3 `latest` is a query

```text
latest = max over discoverable runs of (created_at, run_id)
```

computed at query time over scan ∪ registry, with optional `--status` filters applied from live manifests. There is no `latest.json`: a pointer file whose only honest specification is "a rebuildable hint with a mandatory fallback path" (V2 §9.3) is pure liability once the fallback is fast enough to be the only path. Determinism under parallel runs follows from the total order on `(created_at, run_id)`; nothing can be stale because nothing is stored.

## 10. CLI

Built on the existing argparse adapter in [cli/main.py](../src/nanoquant/cli/main.py):

```bash
nanoquant runs list   [--run-root PATH] [--status S] [--experiment N] [--limit N] [--json]
nanoquant runs show   SELECTOR [--run-root PATH] [--json]
nanoquant runs path   SELECTOR --kind {run,manifest,events,journal,report,log,memory-log} [--run-root PATH]
nanoquant runs rebuild-registry [--run-root PATH] [--include-root PATH]...
nanoquant logs        SELECTOR [--run-root PATH] [--follow] [--level L] [--json] [--memory] [--save] [--poll-seconds F]
```

Selectors, in precedence order:

- exact run ID;
- `latest` (per §9.3);
- `exp:N` — newest `(created_at, run_id)` among runs whose manifest records experiment N; multiple candidates with identical keys or inconsistent manifests are listed and the command exits 2 rather than guessing;
- `--path DIR` — direct directory escape hatch. Works for legacy directories with no manifest at all: `runs show` reports them as `unmanaged`, `logs` renders their `events.jsonl` directly. This is the entire legacy story — no `runs import`, no retroactive writes into old evidence directories.

Behavior:

- `runs list` prints a stable table (run ID, status, created, experiment, component, path) and supports `--json`;
- `runs show` prints normalized manifest metadata and flags missing/torn files;
- `runs path` prints exactly one absolute path with no decoration, for shell composition; `--kind log` and `--kind memory-log` resolve the derived text views, while `--kind journal` is the only sanctioned way to resolve the progress journal, keeping the log/journal vocabulary separated;
- exit codes: `0` success; `2` invalid usage/selector/ambiguity; `3` no matching run; `4` corrupt manifest/registry/event prefix; `5` permission or I/O failure.

All commands are read-only against run directories except `logs --save` (writes `run.log` and `memory.log`) and `rebuild-registry` (rewrites the registry under its lock).

## 11. Diagnostic instrumentation policy

### 11.1 Boundary rule

`domain/factorization.py`, `domain/scale_fit.py`, and planning policies stay side-effect-free: they return typed results, trace samples, and profiler counters. Events are emitted where `StageContext`/`EventSink` already live — application stages and orchestration code. `application/retry_loop.py` emits retry-budget decisions; `ScaleFitStage` emits fit acceptance/rollback from typed diagnostics.

### 11.2 Default info stream

Sufficient to monitor progress from `logs --follow` and memory from `logs --memory --follow`:

- run created/resumed/started/completed/failed/interrupted (plus `lease_taken_over`, `tail_recovered`, `destination_disabled` from this design);
- calibration/plan identity selected or reused;
- block start/completion with index, committed sequence, loss, elapsed seconds;
- layer attempt completion and retry decision with rank, error metrics, thresholds, accepted attempt, budget delta, reason;
- scale-fit accept/rollback with before/after metrics;
- tuning epoch summaries at existing commit/cooldown boundaries only;
- artifact commit references and resume discoveries;
- resource samples at the existing `record_resource_interval_seconds` cadence, routed only to the memory view.

### 11.3 Debug, and what stays out

- `record_admm_steps: true` (with `event_level: debug`, enforced by `OBS003`) enables **sampled** ADMM diagnostics at debug — first, last, every Nth, and anomalous-residual iterations, emitted after the factorizer returns from its pure loop. `false` keeps one aggregate diagnostic per attempt.
- The event stream never duplicates raw phase samples held by the profiling aggregator; it may reference a profile phase or summarize an anomaly. `emit_span_events` remains the profiling design's opt-in.

### 11.4 Performance budgets

- Tiny CPU integration pipeline, default observability: ≤ 2% median wall-time and ≤ 10 ms absolute overhead versus the current sink.
- Representative resident factorization attempt, default info logging: < 1% wall time and **zero added CUDA synchronizations**.
- Debug ADMM sampling gets its own measured, opt-in budget.
- Benchmarks compare the router (JSONL only; JSONL+console) against the current `JsonlEventSink`; rejected or reshaped candidates are recorded in [16-behavior-preserving-optimizations.md](16-behavior-preserving-optimizations.md).

## 12. Lifecycle walkthroughs

**New managed run** — validate config → create directory + initial manifest (atomic) → `open_run_session` (lease, tail check on empty stream, router) → transition running, emit `run.started` → execute → emit terminal event → session close (flush, render `run.log` and `memory.log`) → transition manifest terminal → render reports → release lease in `finally`. The only new steps versus today are the session wrapper and the close-time renders.

**New resident run** — build request → construct resident `RunManifest` (§7.2) → `open_run_session(output, registry_root=...)` which writes the manifest, takes the lease, registers one immutable registry line → proceed as today with `session.events` in place of the raw sink.

**Resume (either kind)** — resolve directory → adopt manifest `run_id` → lease (with stale takeover) → torn-tail recovery → continue sequence → emit `run.resumed` with committed journal sequence → progress-journal discovery unchanged.

**Crash** — no text log was being maintained, so nothing derived is half-written; the registry line was written at start and is immutable; the manifest shows the last transition (`running`); `runs list` shows the run with its manifest status and `runs show` flags a torn event tail if one exists; the next writer quarantines it under the lease. If canonical event writing fails mid-run, `EventWriteError` fails the run and the manifest records a structured failure where possible.

## 13. Deliberate differences from V2

Adopted from V2 unchanged, because the review dictates them and they are right: the router/destination split with single envelope creation, required/optional failure roles, the bounded field contract, the one-line render format, the CLI selector and exit-code scheme, the domain/application boundary rule, and the performance budgets.

Divergences, each with its cost stated:

| # | V2 | V3 | Why | Accepted cost |
|---|---|---|---|---|
| 1 | Writer-maintained `run.log` with reconcile states + `.run-log-state.json` sidecar | Computed rendering; one-shot atomic snapshot at close / `--save` | Deletes an entire reconciliation state machine, its crash windows, and its test matrix; derived-view divergence becomes impossible rather than tested-against | No OS-level `tail -f run.log` during a live run; `logs --follow` is the supported live view |
| 2 | `index.jsonl` with lifecycle snapshots, folding, compaction + `latest.json` pointer + repair paths | Registry of immutable facts for external runs only; managed runs found by scan; `latest` computed | Nothing stored can be stale, so no fold/compact/repair logic exists; the manifest is the only status authority in fact, not just in principle | Listing reads N manifests per invocation (interactive-fast at this project's scale; cache is a documented future option) |
| 3 | Three new config fields + two flush intervals, threshold triangle invariant, config-hash review | Zero new fields; validate and enforce existing `console_level`/`event_level` | Review finding 3.9 (schema/hash churn) dissolves instead of being mitigated; one threshold pair instead of three | Text verbosity is not independently configurable — rendering filters are CLI flags, which is where display concerns belong |
| 4 | Six severities (`trace`, `critical`) | Four (`debug`..`error`) | No producer exists for the extra levels; additive growth later is cheap | Someone adding a `trace` producer later also adds the enum member |
| 5 | Full `RunManifest` "if not too large", else temporary `ResearchRunManifest` adapter with scheduled migration | One manifest schema now — `resolved_config` is already an untyped dict | No dual-schema normalization in catalog readers, no migration debt | Resident manifests carry a request-shaped config dict rather than a `RunConfig`-shaped one; readers treat `resolved_config` as opaque, which they must anyway |
| 6 | `runs import PATH` for unmanaged evidence | `--path DIR` read-only access; no retroactive writes | Old numerical evidence directories are never mutated by tooling | Unmanaged runs don't appear in `runs list` unless a root containing them is scanned or registered |

## 14. Test and acceptance matrix

**Router and destinations** — sequence/timestamp assigned exactly once and shared; `Event | None` contract at each level boundary; all four severities order correctly; `OBS001–003` validation; required-before-optional delivery order; optional-failure quarantine emits `destination_disabled` once without recursion; required failure raises `EventWriteError`; close idempotent across success/failure/interrupt; fault-injected console failure yields byte-identical artifacts and journal for a deterministic fixture.

**Fields** — non-finite floats, oversized strings/events, unsupported types, deep nesting, secret-shaped keys and values (reusing the environment redaction test vectors); deterministic serialization; no `default=str` anywhere in the pipeline.

**Tail and resume** — backward last-sequence scan equals full-scan result; torn tail quarantined to `.orphan` and truncated only under lease; readers report but never modify torn tails; sequence continues correctly across process restarts; legacy constant-run-id streams resume with preserved sequence.

**Sessions and leases** — resident session creates manifest/lease/registry line; resume adopts run ID and warns (not fails) on config-hash drift; stale same-host lease taken over, live or foreign lease raises actionable `RUN001`; registration failure degrades to warning.

**Registry and latest** — concurrent registrations from two processes both survive; torn registry line skipped and repaired by rebuild; `(created_at, run_id)` ordering with and without status filters, under parallel runs; path normalization, run-ID mismatch rejection, traversal records rejected; rebuild idempotent; no argv, environment dump, or secret material in any registry record.

**Rendering and CLI** — one physical line per event under Unicode/newline/control-character torture; run/memory projections partition meter noise while preserving critical diagnostics; `--level` can only subtract; each `--save` output is atomic; close-time renders are present for completed runs and absent (then generatable) for killed runs; follow handles partial lines, truncation, replacement, Ctrl+C, and terminal-status exit on Windows and POSIX; every selector and exit code exercised; `runs path --kind journal` vs `logs` never conflate streams; legacy JSONL-only directory fully inspectable via `--path`.

**Performance** — budget benchmarks of §11.4 with results recorded in Docs/16.

## 15. Implementation phases

Each phase lands independently on the current branch after its tests pass.

1. **Event core.** `Severity` + `OBS0xx` validation; `sanitize_fields` + redaction; `EventRouter` + JSONL/console destinations with failure policy; backward tail scan + torn-tail recovery; `JsonlEventSink` compatibility wrapper; docstring durability fix; overhead benchmark.
2. **Sessions and identity.** `open_run_session`; lease owner metadata + stale takeover; adopt in `bootstrap.run_experiment`; resident manifests and session adoption in `resident_quantization.py` (both entry points) and `tiny_pipeline.py`; close-time `run.log` and `memory.log` renders.
3. **Discovery and CLI.** Registry + lock + rebuild; `runs list/show/path/rebuild-registry`; `logs` with `--follow/--level/--json/--memory/--save`; exit codes; Windows-specific follow/lock tests.
4. **Diagnostic enrichment.** §11.2 info events at application boundaries; sampled ADMM debug behind `record_admm_steps`; budget measurement and Docs/16 records.

Phases 1–2 deliver safety and identity before any new event volume exists; phase 3 delivers the researcher-facing payoff; phase 4 is deliberately last, per the review's scope ordering.

## 16. Rejected alternatives

- **Writer-maintained `run.log`** — rejected above (§8.3, §13.1). The live-follow use case is served by one tested CLI path instead of a second write path with reconciliation semantics.
- **`latest.json` pointer (JSON or symlink)** — rejected; a hint that requires a full fallback implementation is net-negative at this scale, and symlinks are additionally non-portable on Windows.
- **Lifecycle snapshots in the discovery index** — rejected; any stored status is stale the moment a manifest transitions. Immutable pointers + live manifest reads make staleness structurally impossible.
- **Per-destination `event_level` filtering with always-persisted envelopes** — rejected; suppression at creation keeps canonical sequences contiguous and makes the canonical-superset property definitional.
- **Raising on unserializable/non-finite field values in production** — rejected; a diagnostics bug must degrade the diagnostic, not the run. Strictness lives in tests via the same sanitizer's marker outputs.
- **`runs import` writing metadata into legacy evidence directories** — rejected; tooling must never mutate historical numerical evidence. Read-only `--path` access covers inspection.
- **Six-level severity now** — rejected as speculative; the enum grows when a producer exists.
- **Event emission inside pure domain math** — rejected, unchanged from the review: domain returns diagnostics, application emits.

## 17. Definition of done

A researcher can start or resume a managed or resident run and get a real run ID, manifest, and lease; find it with `runs list`/`latest`/`exp:N` from any shell on Windows or POSIX; follow concise progress and memory views independently and read `run.log`/`memory.log` afterward; rebuild the registry and regenerate every rendering from `manifest.json` + `events.jsonl` alone; and turn on bounded debug diagnostics — all without adding a configuration field, changing `config_hash`, altering `state/journal.jsonl` semantics, adding a CUDA synchronization, or making it possible for a derived view to disagree with canonical evidence.
